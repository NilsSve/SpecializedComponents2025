Com ActiveX Samples

 

These samples show you how you can import and use ActiveX controls in your VDF applications. We’ve imported several common ActiveX controls and placed them in views. In all cases, these controls are imported using the IDE’s Import ActiveX Control option from the Tools Menu.

 

We have created the following examples:

 

  • A popup calendar selection panel.
  • An Order entry view with a calendar that can be used for data entry and display
  • A view that lets you select and view or listen to any media player file (wav, mpg, mp3, etc.)
  • A view with an embedded Internet Explorer browser
  • A view that allows you to connect to and work with an Automation Control

.

A couple of things to note about this import process:

 

·         When an ActiveX control is imported it is also added to the IDE’s Controls Palette. In this sample we’ve added all of these controls to a control page named “ActiveX”. You can add these to any page you like.

·         You can only import ActiveX classes when all IDE components are closed. Adding controls requires that the workspace’s class list get rebuilt. This can only occur when no classes are being used (i.e. there are no open components).

·         When you import a class you are given the choice of placing it within a Global or Local (current workspace) repository. If you want a control to be available to all workspaces, select “Global”. For our sample, all classes have been placed in the current workspace area. While you are experimenting with this feature, you will probably want to place your controls in the Local area. Pay attention to this setting – it defaults to Local.

·         A checkbox determines if an imported control should be data aware. Most controls will not be data-aware. You can use these controls to interact with your database but the interactions will have to be manually programmed. Making a control data aware gives it all of the capabilities of a VDF DEO object. The control will automatically interact with the data dictionaries just like the standard DEO classes do (e.g. dbForm).

o        You can create a data aware control if the control can be used to represent a single data field value and the control supports the ability to get and set the value, and provides some kind of event notification when the control’s value is changed.

o        When you create a data aware control, you need to complete a few “ToDo” items in the class before you can compile and use it. These usually involve simple changes. Basically, you need to tell the control how it should “shake hands” with DDO/DEO interface.

o        We’ve created a data aware calendar control (cComDbCalendar). You can look at this to see how this is done. We have also created a non data aware Calendar control (cComCalendar). Although the control is not data aware, it is used in a data entry context – it is used to create a popup calendar selection panel.

·         When a class is imported, the following occurs:

o        The class package is created and placed in either Visual DataFlex 8.3\pkg (if global) or in the workspace’s AppSrc directory (if local).

o        A Bitmap file (bmp) and IDE Meta Class definition file (dfc) are placed in the IdeSrc directory. If global it is placed in usr\global\idesrc; if local, it is placed in the workspace IdeSrc directory. Non data aware classes also generate an object preference file (dfo) and place it in the appropriate IdeSrc directory.

o        An entry for the class is added to the IDE’s class list.

·         You can remove an imported class by carefully:

o        Deleting the class from the IDE’s class list (Tools / Maintain Classlist…)

o        Deleting the pkg file from the appropriate pkg or AppSrc area.

o        Deleting the dfc, bmp and dfo file from the appropriate IdeSrc area.

o        You could also use this technique to move classes between the global and local areas.

 

All of the samples used here were imported within the IDE. Only one class was actually changed after import. The data aware cComDbCalendar class required customization. In addition, we trimmed the interface in this one class. Most of the time you will find that you only need to use a few methods supported by an ActiveX control. It is possible to import a class that contains hundreds of messages (or more) and you will only need to use one or two of these methods. If you wish, you can remove most of the unneeded methods from the class definition.

Calendar Popup Example

 

For this sample, we import a non data aware calendar control. The control is placed in a lookup list. This can be used by any promptable VDF form (Form, dbForm, dbGrid) by setting the form’s prompt_object property to the popup’s object ID. This sample shows you how to coordinate moving data into and out of the calendar control. This popup control could be used by any view that needs a popup calendar.

 

Most of the code for this example is found in the lookup component DatePopup.sl.

Order Entry with Data Aware Calendar Control

 

We imported the same calendar control but created a data aware class for this sample. After the class is created we needed to modify a couple of ToDo items in the class code allowing us to connect the ActiveX interface to the DEO interface. This requires changing two methods. Once implemented, this control can be used like any other IDE DEO data entry control. Just drop it in your view and connect it to a file and field.

 

The order entry view contains two entry objects for the same date field – a date entry spinform and the calendar control. You don’t need two controls. The calendar control could be used standalone. We used both controls to show how versatile the data aware ActiveX control is. When you change a date value in the spinform, the calendar is updated. When you change the date in the calendar, the spinform is updated. Also, notice that the calendar has all of the features you would expect of a DEO. It supports a prompt key, find keys, a save key and a delete key. It responds properly to tool bar buttons (find, clear, prompt, etc.) and it supports status help. All of this support is built right into the class.

 

Most of the code of interest in this sample is found in the imported class cComDbCalendar in the file cComDbCalendar.pkg. We have also included the unaltered class in the file cComDbCalendar-original.pkg so you can compare the two.

Media Player

 

This is a very simple view that lets you select and play media files. To use this all you have to do is to supply the control a file name. These types of controls can be imported and used within an application within minutes.

Internet Explorer Example

 

This sample embeds the Microsoft Internet Explorer active control inside of a view. We create a form for selecting a URL, a button bar for handling the most common browser functions (forward, back, refresh, etc.) and a status panel at the bottom. The net result is a mini browser inside of a view. If you’ve got the WebApp Studio installed on your machine you can even run a WebApp application inside of your regular VDF application.

 

This is the most complicated of the “Com ActiveX” samples. No changes were made to the browser class cComWebBrowser (cComWebBrowser.pkg). If you look at the class you will see it has a big interface, mostly unused by our sample. This might be a good candidate for “interface trimming”.

Simple Automation Controller Example

 

This sample shows how you can connect to and work with a Simple Automation Control. A good way to see this sample in action is to resize the COM Samples application so it takes up the left half of your screen, then click the 'Connect to Server" button and move the Simple Automation Server that pops up to the other half of the screen, so that the Automation Server and the sample with the Simple Automation Controller view open are side by side.

 

Now you can change the settings, like 'Set Shape' and 'Set Color' and see them immediately take effect in the Simple Automation Server. You can also type text into the form next to the 'Set Text' button and click the 'Set Text' button to set the text displayed in the form in the Simple Automation Server. You can also retrieve the text back from the Automation Server by clicking the 'Get Text' button. Last, you can tell the Automation Server to display all of its current settings by clicking the 'Show Server Info' button.
